# عنوان راهنمای سبک (h1)

برای عنوان اصلی فصل از سرتیترهای h1 (`# Heading Title`) استفاده کنید. تمام سرتیترهای hx در نوار کناری به عنوان نشانک ظاهر می‌شوند.

اگر عنوان یک سرتیتر h1 نباشد، نوار کناری یک خطای تکراری نمایش می‌دهد که صفحه و سپس سرتیتر h2 'عنوان' را نشان می‌دهد.

---
## سرتیترهای بخش (h2)

برای هر بخش اصلی جدید از سرتیترهای h2 (`## Section Heading`) که با یک جداکننده (`---`) شروع می‌شوند، استفاده کنید.

### سرتیترهای زیربخش با افزایش سطح h به ۱

برای هر زیربخش جدید از سرتیترهای h3 (`### Subsection Heading`) و برای هر زیر-زیربخش جدید از سرتیترهای h4 (`#### Sub-Subsection Heading`) استفاده کنید.

### پیشرفت h زیربخش: ۲ > ۳ > ۴

اگر پیشرفت h به درستی رعایت نشود، زیرتیترها در نوار کناری با هم مخلوط می‌شوند.

نشانگر سرتیتر باید یک واحد افزایش یابد یا تغییر نکند.

h2 به دنبال h2 یا h3 خوب است. h3 به دنبال h3 یا h4 نیز خوب است.

### خطاهای زیربخش
h2 به دنبال h4 مشکل ایجاد می‌کند. نوار کناری، زیرتیتر را جایی بالاتر از سرتیتر در فهرست مطالب قرار می‌دهد.

### کاهش سطوح h

این به نظر یک مشکل نیست. مثلاً h4 به h3 یا h2 خوب است.

## قراردادهای قالب‌بندی

سبک‌های زیر برای این SRD‌های _ضروری_ (Imperative) در فرمت Markdown استاندارد هستند.

### پررنگ، کج و جداکننده‌ها

- برای متن **پررنگ** از دو ستاره (`**Bold**`) استفاده کنید.
- برای متن _کج_ از زیرخط (`_Italic_`) استفاده کنید.
  - _نکته: از استفاده از یک ستاره برای متن کج خودداری کنید!_
- برای 'فرار' از یک کاراکتر از بک‌اسلش (`\_`) استفاده کنید: \_متن کج نیست\_.
- برای ساخت `code blocks` از گیومه‌های زاویه‌دار (``` `code` ```) استفاده کنید.
- برای درج یک خط جداکننده افقی از سه خط تیره (`---`) استفاده کنید.

### قراردادها برای پررنگ و کج

هدف این است که به الگوهای زیر پایبند باشید...

#### عناوین پررنگ

برای توصیف جادوها، تأثیرات و غیره از عناوین پررنگ استفاده کنید.
- **عنوان جادو:** توضیحات.

#### عناوین کج

برای 'زیرفهرست‌ها'ی جادوهای مختلف و غیره از عناوین کج استفاده کنید.
- **جادو:** توضیحات.
  - **_نوع جادو-الف:_** توضیحات.
  - **_نوع جادو-ب:_** توضیحات.

#### ارجاعات عنوان کج

هنگام ارجاع به کتابی مانند _Mythras Imperative_ یا مجموعه قوانین اصلی _Mythras_ از عناوین کج استفاده کنید.

## جداول

به جداول یک زیرسرتیتر بدهید تا در نوار کناری به عنوان نشانک ظاهر شوند، با یک سطح تورفتگی از بخش والد خود.

~سطر عنوان را پررنگ کنید.~
پررنگ کردن سطر عنوان در واقع آن را به صورت متن عادی رندر می‌کند. CSS پیش‌فرض آن سطر را پررنگ کرده است.

### تراز متن جدول

برای تراز محتوا از ردیف ۲ استفاده کنید: `:--`، `:-:`، یا `--:`.
این بر کل ستون تأثیر می‌گذارد.

#### طرح‌بندی پایه جداول

| **ستون ۱** | **ستون ۲** |
| :-: | :-- |
| ۱ | آلفا |
| ۲ | بتا |
| ۳ | گاما |

### قراردادهای تراز متن

- برای اعداد یا فهرست‌های مقادیر مشابه از تراز وسط استفاده کنید.
- برای جملات متنی از تراز چپ استفاده کنید.

#### جدول مثال تراز

| **عدد** | **توضیحات** |
| :-: | :-- |
| ۱ | متن توصیفی |
| ۲ | متن توصیفی |
| ۳ | متن توصیفی |

#### مثال: جدول شدت آتش

| **شدت** | **مثال‌ها** | **زمان برای شعله‌ور شدن** | **آسیب** |
| :-: | :-- | :-: | :-: |
| ۱ | شمع | ۱d۴ | ۱d۲ |
| ۲ | مشعل | ۱d۳ | ۱d۴ |
| ۳ | آتش کمپ | ۱d۲ | ۱d۶ |
| ۴ | آتش‌سوزی فراگیر اتاق | ۱d۲ | ۲d۶ |
| ۵ | گدازه آتشفشانی | فوری | ۳d۶ |

## پیوندها

پیوندها در SRD کاربردهای بسیاری دارند و در اینجا تنها چند مورد اصلی پوشش داده شده است.

### قالب‌بندی پیوندها

قالب‌بندی یک پیوند با متن داخل براکت انجام می‌شود. برای متن پیوند قابل مشاهده از براکت‌های مربعی و برای آدرس URL مقصد از براکت‌های منحنی استفاده کنید: `[Link Text](URL)`.

در این اسناد، متداول‌ترین پیوند خارجی [Mythras Encounter Generator](https://mythras.skoll.xyz) خواهد بود.
```
[Mythras Encounter Generator](https://mythras.skoll.xyz)
```

### پیوندهای خارجی در مقابل داخلی

پیوندهای خارجی از طریق اینترنت به صفحات دیگر می‌روند و به یک URL `http://example.com` نیاز دارند و به طور پیش‌فرض، _همیشه_ در یک تب مرورگر جدید باز می‌شوند.

برای جلوگیری از شلوغ شدن سند در حال مطالعه با تب‌های متعدد، پیوندهای داخلی و نشانک‌ها با URL‌های نسبی که به فایل‌های Markdown قابل مشاهده در مخزن GitHub ارجاع می‌دهند، تشکیل می‌شوند. به جای یک تب جدید، این پیوندهای داخلی به سرعت به بخش مورد نظر می‌روند.

#### مثال‌های پیوندهای داخلی

- **پیوندهای فصل:** اینها به ابتدای یک بخش اصلی (h1) اشاره می‌کنند. مثلاً فصل [Combat](0005_Combat.md).

  - فرمت پیوند: `[Combat](0005_Combat.md)`

- **پیوندهای زیربخش:** اینها به یک سرتیتر در یک فصل اشاره می‌کنند که سند را مانند بالا ارجاع می‌دهد اما `?id=` و بلافاصله پس از آن سرتیتر مقصد را اضافه می‌کند. مثلاً [Leaping Attacks](0005_Combat.md?id=leaping-attacks)

  - فرمت پیوند: `[Leaping Attacks](0005_Combat.md?id=leaping-attacks)`

#### پیوندهای خارجی به زیربخش‌ها

ممکن است occasionsی برای ارجاع به بخش شناخته شده‌ای از یک سند خارجی وجود داشته باشد، به عنوان مثال، یک جدول خاص در یک SRD دیگر. قالب‌بندی همانند پیوندهای زیربخش بالا است، فقط با استفاده از URL خارجی HTTP.

به عنوان مثال، در اینجا [جدول تبدیل‌های امپریال به متریک](https://cfi-srd.mythras.net/#/Appendix_B_Conversion_Tables?id=imperial-weights-and-measures-to-metric-conversion) از [Classic Fantasy Imperative SRD](https://cfi-srd.mythras.net/) آمده است.

```
[Imperial to Metric conversions table](https://cfi-srd.mythras.net/#/Appendix_B_Conversion_Tables?id=imperial-weights-and-measures-to-metric-conversion)
```

## فهرست‌های جادو و قدرت‌ها

یک بخش 'تمیز' را با یک سرتیتر h2 برای فهرست‌های جادوها، قدرت‌ها و غیره شروع کنید؛ سپس h3 برای هر آیتم فهرست.

جادوها و قدرت‌ها معمولاً در یک فهرست مسطح واحد بر اساس نوع جادو (جادوی عامیانه، خداباوری و غیره) ارائه می‌شوند، اما اگر فهرست به زیربخش‌های خاص خود نیاز داشته باشد، مانند تخصص‌های مدرسه یا فرقه، سطح سرتیتر را به طور مناسب افزایش دهید. به عنوان مثال، h2 برای فهرست معجزات خداباوری، h3 برای فرقه: میترا، و h4 برای هر معجزه فرقه میترا.

به ویژه در مورد جادوها، اما در هر توضیحی که از ویژگی‌های کلمه‌کلیدی استفاده می‌کند، آنها را به صورت متن کج فهرست کنید.

متن توصیفی و قوانین خاص از متن عادی استفاده می‌کنند.

### قالب‌بندی فهرست جادو

```
## h2 List name

### List item

_Attributes_

Description text.
```

## بلوک‌های آماری وسایل نقلیه
یک بخش 'تمیز' را با یک سرتیتر h2 برای فهرست‌ها شروع کنید، سپس سطح h را طبق معمول برای دسته‌ها و آیتم‌های فهرست افزایش دهید.

هر مثال فهرست شده دارای آمار و توضیحات به صورت یک فهرست نقطه‌گذاری شده با عناوین نقطه‌ای پررنگ است.

### قالب‌بندی بلوک آماری وسیله نقلیه

```
## h2 List name

### Category

#### List Item

- **Stat 1:**  [Value]

- **Stat 2:**  [Value]

- **Description:** Descriptive text.
```

## بلوک‌های آماری موجودات
یک بخش 'تمیز' را با یک سرتیتر h2 برای فهرست‌ها شروع کنید. بلوک‌های آماری موجودات از جداول استفاده می‌کنند و از نظر قالب‌بندی و الزامات فضا پیچیده‌تر هستند. برای وضوح، از زیربخش‌ها خودداری کنید و بین آیتم‌های فهرست جداکننده اضافه کنید.

هر مثال فهرست شده باید شامل موارد زیر باشد...

- **عنوان موجود:** یک جداکننده و سپس یک سرتیتر h3 برای نام موجود.

- **ورودی MEG:** یک پیوند خارجی به موجود مشخص شده در [Mythras Encounter Generator](https://mythras.skoll.xyz/).

- **جزئیات اصلی:** نام/نوع موجود، ویژگی‌ها، مهارت‌ها و هر توصیف خاص یا یادداشت‌های شخصیتی در صورت نیاز.

- **جزئیات سلاح:** توضیحات کوتاه شده با ستون‌هایی برای نام سلاح و ویژگی‌های زیر: اندازه (شامل نیرو در اینجا)، برد (شامل محدوده در اینجا)، آسیب، AP/HP، و هر قانون خاص.

- **مکان‌های ضربه و زره:** جداول مکان‌ها بر اساس آناتومی متفاوت موجودات متفاوت خواهد بود اما باید حتی برای انسان‌نماهای سنتی نیز به عنوان وسیله‌ای برای ردیابی زره و آسیب گنجانده شود.

### قالب‌بندی بلوک آماری موجود

```
## Bestiary/List Name

Descriptive text. 

---
### Example Creature
[MEG Entry](https://mythras.skoll.xyz/enemy_template/8626/)

| Creature Name | Example Creature  |
| :-- | :-- |
| Key Attributes | STR 11, CON 14, SIZ 11, DEX 11, INT 11, POW 11, CHA 7 |
| Other notes | Values and details | 
```

## خطاهای جامعه

### اشتباهات تایپی و سردرگمی

ویرایش‌ها برای 'پاکسازی' متن نویسندگان باید حداقل باشد، اما اشتباهات تایپی و سایر خطاها باید به صورت درون خطی و بدون هیچ نشانه‌گذاری تغییر یابند.

### پنل‌های اطلاعات خطا

در مواردی که یک قانون خاص بیش از حد مبهم، ناقص یا به سادگی اشتباه است، یک یادداشت خطای جامعه قابل جمع شدن (collapsible) اضافه کنید.

مسئله را به صورت کج توضیح دهید و فقط برای عنوان مسئله از کج پررنگ استفاده کنید - _بدون سرتیتر h(x)!_ - تا از ایجاد ورودی‌های سردرگم‌کننده در نوار کناری جلوگیری شود.

اگر متن جایگزین اضافه می‌شود، آن را در گیومه دوتایی و به صورت کج پررنگ قرار دهید، که با "_Community Errata:_" به صورت کج با وزن عادی پیش از آن قرار گیرد.

قوانین اضافی، در صورت نیاز، باید به صورت متن با وزن عادی ارائه شوند.

#### الگوی قالب‌بندی خطا

```
<details>

<summary>* Community Errata</summary>

_**Issue Title**_

_Community Errata:_
_**"Replacement text if required."**_

_Explanation text stating how the rule as written causes problems._

</details>
```

<details>

<summary>* خطای جامعه</summary>

## هشدارهای متفرقه

### خطوط خالی / برگشت سطر

بین انواع مختلف ورودی یک خط خالی قرار دهید. سرتیترها، جداکننده‌ها، جداول و غیره. اینها در نمای خواندن قابل مشاهده نخواهند بود اما عدم وجود آنها می‌تواند مشکلات قالب‌بندی ایجاد کند.

### متن چند پاراگرافی پررنگ یا کج

هر پاراگراف باید به عنوان متن پررنگ یا کج علامت‌گذاری شود. خط جدید بین پاراگراف‌ها قالب‌بندی را از بین می‌برد و نشانه‌های قالب‌بندی یتیم باقی می‌گذارد.

### املاهای بین‌المللی

قدرت‌های ارتجاعی و ستمگر صلاح دیده‌اند که تمام املاهای بریتانیایی را به نفع نسخه‌های ضعیف‌تر آمریکایی حذف کنند. به دنبال کلمات بریتانیایی -OUR باشید: _armour_، _favour_ و غیره.
