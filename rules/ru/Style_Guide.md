# Руководство по стилю (h1)

Используйте заголовки h1 (`# Heading Title`) для основного названия главы. Все заголовки hx будут отображаться на боковой панели как закладки.

Если заголовок не является заголовком h1, на боковой панели будет ошибка дублирования, показывающая страницу, а затем заголовок h2 «title».

---
## Заголовки разделов (h2)

Используйте заголовки h2 (`## Section Heading`), предваряемые разделителем (`---`), для каждого нового основного раздела.

### Заголовки подразделов, увеличивающие уровень h на 1

Используйте заголовки h3 (`### Subsection Heading`) для каждого нового подраздела и заголовки h4 (`#### Sub-Subsection Heading`) для каждого нового под-подраздела.

### Прогрессия h-уровня подразделов: 2 > 3 > 4

Подзаголовки путаются на боковой панели, если прогрессия h-уровня не соблюдена правильно.

Маркер заголовка должен увеличиваться на единицу или оставаться без изменений.

h2, за которым следует h2 или h3, допустимо. h3, за которым следует h3 или h4, также допустимо.

### Ошибки подразделов
h2, за которым следует h4, вызывает проблемы. Боковая панель помещает подзаголовок где-то над заголовком в оглавлении.

### Уменьшение h-уровней

Это, кажется, не является проблемой. Например, переход от h4 к h3 или h2 допустим.

## Соглашения по форматированию

Следующие стили являются нормой для данных SRD _Imperative_ в формате Markdown.

### Жирный шрифт, курсив и разделители

- Используйте двойные звёздочки (`**Bold**`) для **жирного** текста.
- Используйте нижние подчеркивания (`_Italic_`) для _курсивного_ текста.
   - _nb: Избегайте использования одинарных звёздочек для курсива!_
- Используйте обратные слэши (`\_`) для «экранирования» символа: \_Не курсив\_.
- Используйте угловые кавычки (``` `code` ```), чтобы сделать `code blocks`.
- Используйте три дефиса (`---`) для вставки горизонтальной разделительной линии.

### Соглашения для жирного шрифта и курсива

Старайтесь придерживаться следующих шаблонов...

#### Жирные заголовки

Используйте жирные заголовки при описании заклинаний, эффектов и т.д.
- **Название магии:** Описание.

#### Курсивные заголовки

Используйте курсивные заголовки для «подсписков» вариантов заклинаний и т.п.
- **Магия:** Описание.
  - **_Вариант магии-А:_** Описание.
  - **_Вариант магии-Б:_** Описание.

#### Ссылки на заголовки курсивом

Используйте курсивные заголовки при ссылке на книгу, такую как _Mythras Imperative_ или основные правила _Mythras_.

## Таблицы

Присваивайте таблицам подзаголовок, чтобы они отображались в боковой панели как закладки, с отступом на один уровень от родительского раздела.

~Сделайте строку заголовка жирным шрифтом.~
Выделение строки заголовка жирным шрифтом фактически отображает её обычным текстом. По умолчанию CSS делает эту строку жирной.

### Выравнивание текста в таблицах

Используйте строку 2 для выравнивания содержимого: `:--`, `:-:` или `--:`.
Это повлияет на весь столбец.

#### Основная структура таблиц

| **Колонка 1** | **Колонка 2** |
| :-: | :-- |
| 1 | альфа |
| 2 | бета |
| 3 | гамма |

### Соглашения по выравниванию текста

- Используйте выравнивание по центру для чисел или списков схожих значений.
- Используйте выравнивание по левому краю для текстовых предложений.

#### Пример таблицы с выравниванием

| **Номер** | **Описание** |
| :-: | :-- |
| 1 | Описательный текст |
| 2 | Описательный текст |
| 3 | Описательный текст |

#### Пример: Таблица интенсивности огня

| **Интенсивность** | **Примеры** | **Время возгорания** | **Урон** |
| :-: | :-- | :-: | :-: |
| 1 | Свеча | 1d4 | 1d2 |
| 2 | Факел | 1d3 | 1d4 |
| 3 | Костёр | 1d2 | 1d6 |
| 4 | Огненная буря, заполняющая комнату | 1d2 | 2d6 |
| 5 | Вулканическая лава | Мгновенно | 3d6 |

## Ссылки

Существует множество применений ссылок в SRD, и здесь рассмотрены лишь несколько основных случаев.

### Форматирование ссылок

Форматирование ссылки выполняется с помощью текста в скобках. Используйте квадратные скобки для видимого текста ссылки и круглые скобки для целевого URL: `[Link Text](URL)`.

В этих документах наиболее часто используемой внешней ссылкой будет [Mythras Encounter Generator](https://mythras.skoll.xyz).
```
[Mythras Encounter Generator](https://mythras.skoll.xyz)
```

### Внешние и внутренние ссылки

Внешние ссылки ведут на другие страницы через интернет и требуют `http://example.com` URL, и по умолчанию они _всегда_ открываются в новой вкладке браузера.

Чтобы избежать множества вкладок, засоряющих читаемый документ, внутренние ссылки и закладки формируются с помощью относительных URL, ссылающихся на файлы Markdown, видимые в репозитории GitHub. Вместо новой вкладки эти внутренние ссылки быстро прокручивают до связанного раздела.

#### Примеры внутренних ссылок

- **Ссылки на главы:** Они указывают на начало основного раздела (h1). Например, глава [Бой](0005_Combat.md).

  - Формат ссылки: `[Combat](0005_Combat.md)`

- **Ссылки на подразделы:** Они указывают на заголовок внутри главы, ссылаясь на документ, как указано выше, но добавляя `?id=`, за которым сразу следует целевой заголовок. Например, [Прыжковые атаки](0005_Combat.md?id=leaping-attacks)

  - Формат ссылки: `[Leaping Attacks](0005_Combat.md?id=leaping-attacks)`

#### Внешние ссылки на подразделы

Может возникнуть необходимость сослаться на известный раздел внешнего документа, например, на конкретную таблицу в другом SRD. Форматирование такое же, как и для ссылок на подразделы, указанных выше, только используется внешний HTTP URL.

В качестве примера, вот [таблица перевода имперских единиц в метрические](https://cfi-srd.mythras.net/#/Appendix_B_Conversion_Tables?id=imperial-weights-and-measures-to-metric-conversion) из [Classic Fantasy Imperative SRD](https://cfi-srd.mythras.net/).

```
[Imperial to Metric conversions table](https://cfi-srd.mythras.net/#/Appendix_B_Conversion_Tables?id=imperial-weights-and-measures-to-metric-conversion)
```

## Списки магии и способностей

Начните «чистый» раздел с заголовка h2 для списков доступных заклинаний, способностей и т.п.; затем h3 для каждого пункта списка.

Заклинания и способности обычно представлены в виде единого плоского списка по типу магии (Народная магия, Теизм и т.д.), но если список требует своих собственных подразделов, таких как специализации школ или культов, повышайте уровень заголовка соответствующим образом. Например, h2 для Списка Чудес Теизма, h3 для Культа: Митра, и h4 для каждого чуда культа Митры.

Особенно в случае заклинаний, но также в любом описании, использующем ключевые атрибуты, перечисляйте их курсивом.

Описательный текст и специальные правила используют обычный текст.

### Форматирование списка заклинаний

```
## h2 List name

### List item

_Attributes_

Description text.
```

## Блоки характеристик существ

Начните «чистый» раздел с заголовка h2 для списков. Блоки характеристик существ используют таблицы и более сложны с точки зрения форматирования и требований к пространству. Для ясности избегайте подразделов и добавляйте разделители между элементами списка.

Каждый перечисленный пример должен содержать следующее...

- **Название существа:** Разделитель, за которым следует заголовок h3 с названием существа.

- **Запись MEG:** Внешняя ссылка на указанное существо в [Mythras Encounter Generator](https://mythras.skoll.xyz/).

- **Основные детали:** Название/тип существа, атрибуты, навыки и любые специальные описания или заметки о персонаже по мере необходимости.

- **Детали оружия:** Сокращенные описания со столбцами для названия оружия и следующих атрибутов: Размер (включите сюда Силу), Досягаемость (включите сюда Дальность), Урон, БР/ОЖ и любые особые правила.

- **Зоны попадания и броня:** Таблицы зон будут различаться в зависимости от анатомии существа, но должны быть включены даже для традиционных гуманоидов как средство отслеживания брони и урона.

## Исправления сообщества

### Опечатки и путаница

Правки для «очистки» авторского текста должны быть минимальными, но опечатки и другие ошибки следует просто исправлять по месту без каких-либо указаний.

### Информационные панели исправлений

Если данное правило слишком двусмысленно, неполно или просто неверно, добавьте сворачивающуюся заметку «Исправления сообщества».

Объясните проблему курсивом, используя только _жирный курсив_ для названия проблемы — _без заголовков h(x)!_ — чтобы не создавать путающих записей на боковой панели.

Если добавляется замещающий текст, поместите его в двойные кавычки _жирным курсивом_, предваряя его "_Исправления сообщества:_" обычным курсивом.

Дополнительные правила, если требуются, должны быть представлены обычным текстом.

#### Шаблон форматирования исправлений

```
<details>

<summary>* Community Errata</summary>

_**Issue Title**_

_Community Errata:_
_**"Replacement text if required."**_

_Explanation text stating how the rule as written causes problems._

</details>
```

#### Примеры исправлений

<details>

<summary>* Исправления сообщества</summary>

_**Лазание**_

_Исправления сообщества:_
_**"Носимая броня препятствует лазанию на величину, равную среднему значению носимых Очков Брони. Чтобы найти это значение, суммируйте очки брони каждой зоны и разделите эту сумму на количество зон."**_

_Правило в текущей формулировке быстро вызывает проблемы. Ношение одной части брони с 6 ОЗП уменьшило бы перемещение человека до 0, что означает, что человек в стальном шлеме и без ничего другого был бы слишком перегружен, чтобы вообще лазать. Это кажется необоснованно ограничительным, отсюда и предложенная выше отредактированная версия. Это будет обновлено, если первоначальные авторы пересмотрят правило в его текущем виде._

</details>

_**Аллюры: Ходьба, Бег, Спринт**_

_Опубликованная версия _Mythras Imperative_ не содержит явных правил для перемещения различными аллюрами, хотя они и охвачены в основной книге правил Mythras. Указанные выше множители аллюров были добавлены как часть Исправлений сообщества._

## Различные предупреждения

### Пустые строки / Возврат каретки

Оставляйте пустую строку между различными типами записей. Заголовки, разделители, таблицы и т.д. Они не будут видны в режиме чтения, но их отсутствие может вызвать проблемы с форматированием.

### Жирный или курсивный многопараграфный текст

Каждый абзац должен быть помечен как жирный или курсивный текст. Новая строка между абзацами нарушит форматирование, оставив разрозненные маркеры форматирования.

### Международные орфографии

Регрессивные и прискорбные силы сочли уместным удалить все британские написания в пользу объективно более слабых американских версий. Обращайте внимание на британские слова с окончанием -OUR: _armour_ (броня), _favour_ (одолжение) и т.д.
