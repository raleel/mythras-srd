# Назва посібника зі стилю (h1)

Використовуйте заголовки h1 (`# Heading Title`) для назви головного розділу. Усі заголовки hx відображатимуться на бічній панелі як закладки.

Якщо назва не є заголовком h1, на бічній панелі виникне помилка дублювання, що показуватиме сторінку, а потім заголовок h2 «назва».

---
## Заголовки розділів (h2)

Використовуйте заголовки h2 (`## Section Heading`), яким передує роздільник (`---`), для кожного нового головного розділу.

### Заголовки підрозділів зі збільшенням рівня h на 1

Використовуйте заголовки h3 (`### Subsection Heading`) для кожного нового підрозділу, а заголовки h4 (`#### Sub-Subsection Heading`) для кожного нового під-підрозділу.

### Прогресія h-рівнів підрозділів: 2 > 3 > 4

Підзаголовки плутаються на бічній панелі, якщо прогресія h-рівнів не обробляється коректно.

Маркер заголовка повинен збільшуватися на одиницю або не змінюватися.

h2, за яким слідує h2 або h3, є прийнятним. h3, за яким слідує h3 або h4, також є прийнятним.

### Помилки підрозділів
h2, за яким слідує h4, викликає проблеми. Бічна панель розміщує підзаголовок десь над заголовком у змісті.

### Зменшення h-рівнів

Це, схоже, не є проблемою. Наприклад, h4 до h3 або h2 є прийнятним.

## Умовні позначення форматування

Наступні стилі є нормою для цих _Імперативних_ SRD у форматі Markdown.

### Жирний, курсив та роздільники

- Використовуйте подвійні зірочки (`**Bold**`) для **жирного** тексту.
- Використовуйте підкреслення (`_Italic_`) для _курсивного_ тексту.
  - _nb: Уникайте використання одинарних зірочок для курсиву!_
- Використовуйте зворотні скісні риски (`\_`), щоб «екранувати» символ: \_Не курсив\_.
- Використовуйте кутові лапки (``` `code` ```), щоб зробити `code blocks`.
- Використовуйте три дефіси (`---`), щоб вставити горизонтальну розділювальну лінію.

### Умовні позначення для жирного та курсивного тексту

Прагніть дотримуватися наступних шаблонів...

#### Жирні заголовки

Використовуйте жирні заголовки для опису заклинань, ефектів тощо.
- **Магічна Назва:** Опис.

#### Курсивні заголовки

Використовуйте курсивні заголовки для "підсписків" варіантів заклинань тощо.
- **Магія:** Опис.
  - **_Варіант Магії-А:_** Опис.
  - **_Варіант Магії-Б:_** Опис.

#### Посилання на курсивні заголовки

Використовуйте курсивні заголовки при посиланні на книгу, таку як _Mythras Imperative_ або основні правила _Mythras_.

## Таблиці

Дайте таблицям підзаголовок, щоб вони відображалися як закладки на бічній панелі, з відступом на один рівень від батьківського розділу.

~Зробіть рядок заголовка жирним.~
Зробити рядок заголовка жирним насправді відображає його звичайним текстом. Стандартний CSS має цей рядок жирним.

### Вирівнювання тексту в таблиці

Використовуйте рядок 2 для вирівнювання вмісту: `:--`, `:-:` або `--:`.
Це вплине на весь стовпець.

#### Основний макет таблиць

| **Стовпець 1** | **Стовпець 2** |
| :-: | :-- |
| 1 | альфа |
| 2 | бета |
| 3 | гамма |

### Умовні позначення для вирівнювання тексту

- Використовуйте вирівнювання по центру для чисел або подібних списків значень.
- Використовуйте вирівнювання по лівому краю для речень тексту.

## Посилання

Існує багато способів використання посилань у SRD, і тут розглядається лише кілька основних випадків.

### Форматування посилань

Форматування посилання виконується за допомогою тексту в дужках. Використовуйте квадратні дужки для видимого тексту посилання та круглі дужки для URL-адреси призначення: `[Link Text](URL)`.

У цих документах найбільш часто використовуваним зовнішнім посиланням буде [Генератор зустрічей Mythras](https://mythras.skoll.xyz).
```
[Mythras Encounter Generator](https://mythras.skoll.xyz)
```

### Зовнішні та внутрішні посилання

Зовнішні посилання ведуть на інші сторінки через Інтернет і потребуватимуть URL-адреси `http://example.com` і, за замовчуванням, _завжди_ відкриватимуться в новій вкладці браузера.

Щоб уникнути захаращення документа кількома вкладками, внутрішні посилання та закладки формуються за допомогою відносних URL-адрес, що посилаються на файли Markdown, видимі в репозиторії GitHub. Замість нової вкладки ці внутрішні посилання швидко прокручуватимуться до пов'язаного розділу.

#### Приклади внутрішніх посилань

- **Посилання на розділи:** Вони вказують на початок основного (h1) розділу. Наприклад, розділ [Бій](0005_Combat.md).

  - Формат посилання: `[Combat](0005_Combat.md)`

- **Посилання на підрозділи:** Вони вказують на заголовок у розділі, посилаючись на документ, як зазначено вище, але додаючи `?id=`, а потім одразу заголовок призначення. Наприклад, [Атаки стрибком](0005_Combat.md?id=leaping-attacks)

  - Формат посилання: `[Leaping Attacks](0005_Combat.md?id=leaping-attacks)`

#### Зовнішні посилання на підрозділи

Може виникнути потреба посилатися на відомий розділ зовнішнього документа, наприклад, на певну таблицю в іншому SRD. Форматування таке ж, як і для посилань на підрозділи вище, але використовується зовнішній HTTP-URL.

Як приклад, ось [Таблиця перетворення імперських одиниць на метричні](https://cfi-srd.mythras.net/#/Appendix_B_Conversion_Tables?id=imperial-weights-and-measures-to-metric-conversion) з [Classic Fantasy Imperative SRD](https://cfi-srd.mythras.net/).

```
[Imperial to Metric conversions table](https://cfi-srd.mythras.net/#/Appendix_B_Conversion_Tables?id=imperial-weights-and-measures-to-metric-conversion)
```

## Списки магії та здібностей

Розпочніть "чистий" розділ із заголовком h2 для списків доступних заклинань, здібностей тощо; потім h3 для кожного елемента списку.

Заклинання та здібності зазвичай представлені єдиним плоским списком за типом магії (Народна Магія, Теїзм тощо), але якщо список потребує власних підрозділів, таких як шкільні або культові спеціалізації, збільшуйте рівень заголовка відповідно. Наприклад, h2 для Списку Чудес Теїзму, h3 для Культ: Мітрас, і h4 для кожного чуда культу Мітраса.

Особливо у випадку заклинань, але в будь-якому описі, що використовує ключові атрибути, перераховуйте їх курсивом.

Описовий текст та спеціальні правила використовують звичайний текст.

### Форматування списку заклинань

```
## h2 List name

### List item

_Attributes_

Description text.
```

## Блоки характеристик транспортних засобів
Розпочніть "чистий" розділ із заголовком h2 для переліків, потім збільшуйте рівень h-заголовка, як зазвичай, для категорій та елементів списку.

## Блоки характеристик істот
Розпочніть "чистий" розділ із заголовком h2 для списків. Блоки характеристик істот використовують таблиці і є складнішими з точки зору форматування та вимог до простору. Для чіткості уникайте підрозділів та додавайте роздільники між елементами списку.

Кожен перелічений приклад повинен мати наступне...

- **Назва істоти:** Роздільник, за яким слідує заголовок h3 для імені істоти.

- **Запис MEG:** Зовнішнє посилання на зазначену істоту в [Mythras Encounter Generator](https://mythras.skoll.xyz/).

- **Основні деталі:** Ім'я/тип істоти, атрибути, навички та будь-які спеціальні описи або нотатки про персонажа за потребою.

- **Деталі зброї:** Скорочені описи з колонками для назви зброї та наступних атрибутів: Розмір (включно з Силою), Досяжність (включно з Діапазоном), Пошкодження, ББ/ОЗ, та будь-які спеціальні правила.

- **Місця влучання та обладунки:** Таблиці місць влучання змінюватимуться залежно від різної анатомії істот, але повинні бути включені навіть для традиційних гуманоїдів як засіб відстеження обладунків та пошкоджень.

```
| :-- | :-- |
| Key Attributes | STR 11, CON 14, SIZ 11, DEX 11, INT 11, POW 11, CHA 7 |
| Other notes | Values and details | 
```

## Виправлення спільноти

### Друкарські помилки та плутанина

Редагування тексту авторів для "приведення до ладу" має бути мінімальним, але друкарські помилки та інші помилки слід просто змінювати безпосередньо в тексті без будь-яких позначок.

### Інформаційні панелі виправлень

Якщо певне правило є занадто неоднозначним, неповним або просто неправильним, додайте згортальну примітку "Виправлення Спільноти".

Поясніть проблему курсивом, використовуючи лише **жирний курсив** для заголовка проблеми – _без заголовків h(x)!_ – щоб не створювати заплутаних записів на бічній панелі.

Якщо додається замінюючий текст, розмістіть його в подвійних лапках **жирним курсивом**, перед яким має йти "_Виправлення Спільноти:_" звичайним курсивом.

Додаткові правила, якщо вони потрібні, повинні бути представлені звичайним текстом.

#### Шаблон форматування виправлень

```
<details>

<summary>* Community Errata</summary>

_**Issue Title**_

_Community Errata:_
_**"Replacement text if required."**_

_Explanation text stating how the rule as written causes problems._

</details>
```

## Різні попередження

### Порожні рядки / Повернення каретки

Залишайте порожній рядок між різними типами записів. Заголовки, роздільники, таблиці тощо. Вони не будуть відображатися в режимі читання, але їх відсутність може спричинити проблеми з форматуванням.

### Жирний або курсивний багатоабзацний текст

Кожен абзац має бути позначений як жирний або курсивний текст. Новий рядок між абзацами порушить форматування, залишаючи "сирітські" позначки форматування.

### Міжнародні написання

Регресивні та ганебні сили вирішили прибрати всі британські написання на користь об'єктивно слабших американських версій. Будьте уважні до британських слів на -OUR: _armour_, _favour_ тощо.
